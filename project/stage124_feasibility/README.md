# Stage124 — Listing-source feasibility probe v2 (non-authoritative)

Technical feasibility check **only**: can the *initial* listing info for the Stage124
tickers be pulled automatically from **TSETMC** (instrument + daily trade history) and
**Codal** search? This folder is a probe — it does **not** build a verified file, does
**not** touch eligibility, and does **not** overwrite any frozen Stage123/Stage124 artifact.

## Script
`probe_listing_sources_stage124_v2.py` is the only official runner for the Iran probe.

```bash
python project/stage124_feasibility/probe_listing_sources_stage124_v2.py
```

`probe_listing_sources_stage124.py` is legacy/deprecated and must not be used for the
official Stage124 Iran probe artifacts.

## Outputs
- `feasibility_probe_flat_stage124_iran.csv` — one row per pilot ticker, exactly using
  `FLAT_COLUMNS` from `probe_listing_sources_stage124_v2.py`.
- `feasibility_probe_report_stage124_iran.json` — run provenance, egress country,
  masked IP, IP SHA-256, source accessibility, script hash, code commit, Stage124
  template before/after hashes, status counts, and non-authoritative flags.
- `raw_responses_manifest_stage124_iran.csv` — one row per HTTP call, exactly using
  `MANIFEST_COLUMNS` from `probe_listing_sources_stage124_v2.py`.
- `instrument_candidate_history_stage124_iran.csv` — candidate-only history rows for
  ordinary candidate insCodes that require manual review, especially `بوعلی` and `نوری`.
- `raw/tsetmc/` and `raw/codal/` — raw response bodies for TSETMC and Codal calls.

## Egress and privacy
The report stores only the egress country, masked IP, SHA-256 of the IP, and check time.
The raw response from the egress check is not saved in Git because it can expose the full
IP address. No credential, proxy, or full IP address should be committed.

## Current run provenance
The committed Iran probe artifacts are intended to be generated by running v2 from an
Iranian egress. A valid real run must have `egress_country_code = IR`,
`valid_iran_run = true`, and at least one accessible Iranian source. The data remains
**candidate-only** and **not verified**: `verified` stays false for every row,
`is_verified_file` is false, and `touches_eligibility` is false.

## Rules enforced
1. earliest TSETMC `dEven` is stored **only** as candidate first-trade evidence.
2. rights (حق‌تقدم), funds (صندوق) and other non-ordinary instruments are excluded.
3. multiple ordinary insCodes → status `ambiguous_instrument`, **no** forced selection.
4. original + normalized symbol + aliases all preserved; `پی‌پاد` includes `پی‌پاد`,
   `پی پاد`, and `پیپاد` search aliases.
5. endpoint, retrieved_at, HTTP status, duration, content type, response size, raw path,
   parse status, error class/message, and raw SHA-256 are preserved in the manifest.
6. `network_unreachable` is kept separate from `no_instrument_match`.
7. Codal search records candidate notices for listing-related keywords.
8. no verified file is produced, eligibility is never changed.

## extraction_status vocabulary
`candidate_found · network_unreachable · no_instrument_match · ambiguous_instrument ·
empty_trade_history · codal_no_candidate_notice · http_error · parse_error · partial_success`

> Pilot-matching notes: `اردستان`, `اروند`, and `سآبیک` candidate dates may conflict with
> prior project information and must remain unverified. `بوعلی` and `نوری` can have multiple
> candidate insCodes; none is force-selected. `اپال` has no company name in the frozen
> template. All rows remain non-authoritative until separately reviewed.
